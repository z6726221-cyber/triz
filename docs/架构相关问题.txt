## 问题一：M1-M3 可以合并为一个 Skill（问题建模 Skill）吗？

**明确结论：可以，而且更符合主流 Agent Skill 设计理念。**

**理由如下：**

1. **从 Skill 的定义出发**  
   Skill 本质是 **“完成一个完整任务的 SOP 说明书”**。对于“问题建模”这个用户可见节点而言，其内部 M1（功能建模）、M2（根因分析）、M3（问题定型）共同完成一个子目标：**将原始用户输入转化为结构化的矛盾描述**。这三个步骤存在明确的数据依赖（M2 依赖 M1，M3 依赖 M2），且中间结果（如 SAO、因果链）仅用于内部流转，对外部用户不需要单独暴露。将这些步骤打包成一个 Skill，恰恰符合 Skill 作为“可插拔能力单元”的定位。

2. **与主流 Skill 规范对比**  
   Anthropic 的 Agent Skills 规范中，一个 Skill 的 `SKILL.md` 可以包含**多个步骤的 SOP**，可以调用多个 Tools，也可以包含多个子步骤的 LLM 推理。例如，一个“代码审查 Skill”可能包含“静态分析 → 复杂度计算 → 生成建议”三个内部步骤，但它们仍然是同一个 Skill。你当前的 M1~M3 完全可以写在一个 `SKILL.md` 里，按顺序执行。

3. **保留边界 ≠ 必须拆分 Skill**  
   架构文档中“对外合并，对内保留边界”的原则是正确的。合并为单个 Skill 并不要求删除 M1/M2/M3 作为独立函数的代码边界，你仍然可以在 Python 实现中保留 `m1_modeling()`、`m2_causal_analysis()`、`m3_formulation()` 三个函数，然后在 `SKILL.md` 中定义它们的调用顺序、失败重试、fallback 逻辑。这种**内部函数拆分 + 外部 Skill 封装**的做法更干净。

4. **潜在风险与对策**  
   - **Prompt 长度增加**：合并后 `SKILL.md` 的正文会包含三个阶段的详细指令，可能达到 4k~6k token。这在主流框架中完全可接受（Claude 上下文 200k），且由于是“按需加载”，并不会造成全局浪费。
   - **可控性下降**：如果 M2 失败需要跳过，可在 `SKILL.md` 中用条件判断描述（例如“如果 M1 未提取到 Harmful SAO，则跳过 M2，直接使用 fallback”），由 LLM 遵循指令实现。如果希望确定性控制，可以将门控逻辑写成一个 Tool（如 `should_run_m2`），在编排器中显式调用。

**推荐调整**：将 M1、M2、M3 合并为一个 **`problem_modeling` Skill**，内部仍然使用三个函数，并且保留你设计的门控（M2 默认触发，仅在无 Harmful SAO 时跳过）。这样既简化了编排器的调用结构，又符合主流 Agent Skill 的“一站式能力”风格。

## 问题二：当前系统的编排逻辑、Skill 调用逻辑，与 Claude Code、OpenClaw 等主流 Agent 一样吗？

**明确结论：核心思想一致，但实现层级不同——你的系统是“显式编排器 + 顺序调用”，主流 Agent 是“动态决策 + 循环调用”。**

我们来逐层对比：

### 相似点（底层机制）
- **Skill 封装**：都是将 LLM 密集的推理任务封装为可复用单元，以 `SKILL.md` + 脚本形式存在。
- **Tool 分离**：确定性操作（查表、矩阵、API 调用）都实现为独立的 Tool，由 Agent 按需调用。
- **渐进式披露**：所有主流框架都会在启动时只加载 Skill 的 name/description，需要时才加载正文——你的系统在启动时没有明确表述这一点，但可以通过实现时模仿该机制。
- **跨模块数据传递**：使用结构化对象（你的 Pydantic 模型等价于 JSON 对象），主流 Agent 框架内部也是类似结构。

### 不同点（编排哲学）
| 维度 | 当前 TRIZ 系统 | Claude Code / OpenClaw |
| :--- | :--- | :--- |
| **控制流** | **静态 DAG**：编排器硬编码调用顺序（问题建模 → 矛盾求解 → 跨界检索 → 方案生成 → 方案评估 → 收敛判断）。 | **动态推理**：Agent 根据当前状态和 Skill 描述自主决定下一步调用哪个 Skill/Tool，没有预定义的全局顺序。 |
| **Skill 调用方式** | 编排器直接调用函数（`m1_modeling(ctx)`）。 | Agent 通过提示词生成 `use_skill("skill_name")` 或类似工具调用，然后系统加载对应的 `SKILL.md` 并让 LLM 继续推理。 |
| **循环控制** | 集中式 M7 Tool 判定 CONTINUE/TERMINATE。 | 隐式循环：Agent 在内部推理循环中不断 `think → act → observe`，直到自我判断任务完成或达到限制。 |
| **用户可见节点** | 预定义 5 个节点，固定顺序展示。 | 无“节点”概念，用户看到的是 LLM 逐步输出思考和行动结果。 |

**你的系统本质上是一个“确定性工作流 + 内部 LLM 模块”**，而 Claude Code 等是 **“通用 Agent 循环 + 可插拔 Skills”**。前者适合 TRIZ 这类领域流程固定的场景，后者适合开放式任务（如编程、通用问答）。

**但值得注意的是**：Claude Code 也支持将复杂任务写成“项目级 Skill”，此时 Skill 内部的 SOP 可以是线性的（类似于你的 M1~M3）。区别在于：**在 Claude Code 中，Skill 内部的线性步骤是由 LLM 阅读 Markdown 后自主执行的；而在你的系统中，线性步骤是由编排器代码直接调用的**。两种方式各有优劣，你的方式更可控、更高效（无 LLM 调度开销），但扩展性较弱。

**结论**：架构思想一致，但实现层级不同。你的设计更适合**垂直领域流程固定**的 Agent 应用，而不是通用 Agent。

## 问题三：可以只用一个 LLM 作为单 Agent，调用这些 Skill 和 Tool 吗？（双 Agent 也可以，但评估模块单独算）

**明确结论：完全可以。你当前的设计本质上已经是“单 Agent + Skills + Tools”架构。**

### 分析你的当前系统
- 虽然你有多个 Skill（M1, M2, M5, M6），但它们**不是并发运行的不同 Agent**，而是**同一个编排器顺序调用的独立 LLM 调用**。每个 Skill 执行时，编排器向 LLM 发送一个包含该 Skill 专用提示词的请求，拿到结果后继续。
- 这不叫“多 Agent”，这叫**单 Agent 的多轮 LLM 调用**。真正的多 Agent 系统（如 AutoGen、CrewAI）会有独立记忆、独立角色、互相通信的多个 LLM 实例。

### 与 Claude Code 的对比
- **Claude Code 是单 Agent**。它内部只有一个 Claude 模型实例，通过系统提示词和上下文管理来调用不同的 Skills/Tools。它并没有为每个 Skill 启动一个新的 Agent 实例。
- **你的系统也是单 Agent**，只不过你是用代码编排器来控制调用顺序，而 Claude Code 是用 LLM 自己的推理结果来驱动下一步。

### 你提到“评估模块可以额外用其他的，双 Agent 也可以”
- 你的 M6 已经设计为**独立 LLM 实例**（temperature=0.1，与 M5 的 0.3 不同），这本质上已经是“双实例单 Agent”模式。你可以继续保留这种做法，它有助于避免评估偏差。
- 如果追求极致简化，也可以让同一个 LLM 实例先后执行 M5 和 M6，但需要在提示词中明确要求“扮演评审者，不得修改方案内容”，并在代码层面强制只读传递。

**核心要点**：**“单 Agent”指的是有一个统一的状态和决策核心，而不是只有一个 LLM 调用**。你的编排器就是那个核心，所有 Skill 和 Tool 都是它的扩展能力。因此，你无需引入多 Agent 框架，当前架构已经是标准的单 Agent 模式。

## 问题四：模块间数据用纯文本（如 Markdown）传输可以吗？如果单 Agent 可行，需要多几次循环吗？

**明确结论：可以用纯文本，但会显著增加解析复杂性和错误率，不推荐。单 Agent 循环次数与数据格式无直接关系，但与任务复杂度和收敛条件相关。**

### 数据格式：JSON vs 纯文本

| 方面 | JSON（结构化） | 纯文本（如 Markdown） |
| :--- | :--- | :--- |
| **LLM 生成稳定性** | 强制格式校验，Pydantic 可自动重试，几乎 100% 可解析。 | LLM 经常输出格式变体（列表没缩进、漏掉标题、用错分隔符），解析失败率高。 |
| **下游模块提取效率** | 直接 `ctx.sao_list` 访问 O(1)。 | 每次需要正则或 LLM 重新提取，增加延迟和 token 消耗。 |
| **确定性模块使用** | Tool 可以直接消费结构化字段。 | Tool 需要先解析文本，引入不确定性。 |
| **可调试性** | Pydantic 校验报错清晰。 | 文本解析失败时难以定位问题。 |
| **对外渲染** | 需要单独渲染函数。 | 可以直接输出，减少一层转换。 |

**结论**：**强烈建议保留 JSON 作为内部传输格式**。你的“对外展示渲染为 Markdown”已经很好解决了用户可读性问题。如果将内部传输也改为纯文本，你会失去结构化带来的稳定性和效率，得不偿失。主流 Agent 内部（包括 Claude Code）也都是用结构化对象传递数据，仅在最终输出或 Skill 的 `SKILL.md` 中用 Markdown 描述。

### 单 Agent 循环次数影响
采用纯文本不会直接影响循环次数。循环主要由 M7 收敛控制决定：
- 如果纯文本导致 LLM 提取信息错误，可能让 M6 评估结果失真，进而导致 M7 误判（比如提前终止或无限循环）。
- 为了弥补解析错误，你可能需要在每个节点前增加**文本清洗与重试循环**（比如调用 LLM 去解析上一节点的 Markdown 输出），这反而会增加 LLM 调用次数。

### 你问“单 Agent 是否需要多几次循环”
- 单 Agent 与循环次数没有必然关系。你的系统已经通过 M7 控制循环，循环次数取决于问题复杂度和理想度提升速度。
- 如果你将当前的“编排器硬编码调用”改为“让单 Agent 自主决策下一步”，那么循环模式会改变：Agent 可能会多次尝试不同的 Skill 顺序，导致循环次数增加。但那是调度策略变化，不是数据格式变化带来的。

**最终建议**：保持 JSON/Pydantic 作为内部传输格式，继续用专用渲染函数输出 Markdown 给用户。这是最稳健、最可维护的设计。

## 问题 1：M1-M3 可以合并为一个 Skill 吗？

**可以，且应该合并。**

### 理由

三个模块全是 LLM 调用，分拆带来的问题比解决的问题多：

**信息折损**：当前 M1 输出 SAO 列表传给 M2，M2 输出 root_param 传给 M3。每次传递都是 Pydantic 序列化后的字段摘取，原始推理上下文被丢弃。M3 看到的是"根因参数：接触面积"，但看不到 M2 是如何从因果链推导出这个结论的完整推理。这增加了误判风险。

**冗余开销**：三次 LLM 调用意味着三次独立理解用户问题、三次独立建立语义模型、三次 token 开销。同一个"手术刀磨损"问题在三个模块中被反复编码解码。

**一致性风险**：M1 标注为 Harmful 的功能，M2 的因果链分析可能给出略有偏差的定性，M3 又基于这个偏差做矛盾定型。偏差在分步传递中会被放大。

**合并后的形态**：单一 `problem_modeling` Skill，一次 LLM 调用完成完整推理链。Prompt 中要求 LLM 在输出中按阶段标记（`### 功能建模` / `### 根因分析` / `### 问题定型`），既保证用户可见各阶段结果，又保证推理在统一上下文中连贯完成。

**例外情况**：如果某个阶段需要用不同模型（如 M2 根因分析用更强的推理模型，M1 用轻量模型节省成本），分拆保留灵活性是有价值的。但就当前架构而言，合并的收益大于代价。

## 问题 2：当前系统的编排逻辑与 Claude Code 等主流 Agent 一样吗？

**不一样。**

### 根本差异

| | 本系统（当前） | 主流 Agent（Claude Code 等） |
|:---|:---|:---|
| 流程决策者 | 编排器（Python 硬编码） | LLM 自身 |
| LLM 角色 | 被动执行被调用 | 主动规划并决策 |
| 工具调用触发 | 编排器按固定顺序调 | LLM 判断时机并主动调 |
| 迭代逻辑 | 编排器的 while 循环 | LLM 的思考-行动-观察循环 |

### 具体说明

本系统中，`Orchestrator.run_workflow()` 方法硬编码了执行顺序：

```python
问题建模 → 矛盾求解 → 跨界检索 → 方案生成 → 方案评估 → [M7判定是否循环]
```

每个 Skill 和 Tool 在哪个时刻被调用、以什么顺序被调用，是编排器写死的。LLM 从不做流程决策，它只负责在每次被调用时接收输入、返回结构化输出。

Claude Code 则完全不同：Claude 收到"帮我修复这个 bug"时，它自己决定先读取哪个文件、发现需要查文档、自行调用搜索工具、根据搜索结果决定修改方案。没有一个外部编排器告诉它做什么，它自己就是控制流。

### 本质定义

本系统应被定义为：**嵌入了 LLM 的固定工作流引擎**，而不是自主 Agent。文档中"Agent Skills + Tools"的命名更多是指模块的组织形式（Skill 封装 LLM 推理，Tool 封装确定性计算），不意味着这是一个 LLM 自主决策的 Agent 系统。

## 问题 3：可以改为单 Agent + Skills + Tools 吗？Claude Code 是单 Agent 吗？

**Claude Code 是单 Agent。本系统可以改为单 Agent 架构，且这是值得探索的方向。**

### Claude Code 的单 Agent 模式

Claude Code 是典型的单 Agent：一个 LLM 实例，拥有一组注册的 Tools（Bash 执行、文件读写、搜索等），在 ReAct 循环中自主推理。它没有外部编排器，system prompt 定义了行为边界，工具通过 function calling 暴露给 LLM。

### 本系统改为单 Agent 的形态

```
单 Agent（一个 LLM 实例，主循环）
│
├── 可调用 Skill（独立 LLM 实例，完成特定专业任务）：
│   ├── problem_modeling     — 问题建模（原 M1+M2+M3 合并）
│   ├── solution_generation  — 方案生成（原 M5）
│   └── solution_evaluation  — 方案评估（原 M6，独立实例防自评偏差）
│
└── 可调用 Tool（确定性函数，无 LLM）：
    ├── contradiction_solver     — 矛盾求解（原 M4）
    ├── cross_domain_search      — 跨界检索（原 FOS）
    └── convergence_control      — 收敛控制（原 M7）
```

**Skill 为何保留而非并入 Agent**：

- `problem_modeling`：有完整独立的输入输出 schema，是专业性强的结构化推理任务。作为独立 Skill 可以单独优化 prompt、单独升级模型、单独监控质量。并入 Agent 会让问题建模的推理与流程决策耦合，调试困难。

- `solution_generation`：同样有明确的任务定义和输出结构，独立封装便于方案生成策略的迭代。

- `solution_evaluation`：**必须独立**。生成方案的 LLM 评估自己的方案会产生自我偏袒，必须用独立 LLM 实例（不同 temperature、不同 system prompt）做评审。这是架构文档的核心原则，单 Agent 下也必须保留这个隔离。

**Tool 必须保留的理由**：

- `contradiction_solver`：参数匹配和矩阵查表需要确定性，不能依赖 LLM 的概率性输出。
- `cross_domain_search`：数据库查询和 API 调用是工程操作，与 LLM 无关。
- `convergence_control`：阈值判定逻辑需要硬规则，LLM 不适合做精确数值比较。

### Agent 自身的职责

Agent 不做 TRIZ 专业推理，它的职责是：

1. **流程编排**：决定何时调用哪个 Skill 或 Tool
2. **结果解读**：将结构化返回转化为用户可读的 Markdown
3. **异常处理**：面对 CLARIFY 或空结果时调整策略
4. **上下文管理**：维护会话历史，迭代时注入 feedback

### 代价

- **可控性下降**：Agent 可能跳过步骤或乱序调用
- **成本上升**：每轮推理携带全量上下文，token 消耗增加
- **可观测性下降**：推理过程混在长文本中，调试定位变难

这些代价需要与灵活性收益一起权衡。对于 TRIZ 这种强方法论场景，保留一定程度的流程约束（如在 system prompt 中明确要求必须依次调用各 Skill/Tool）是折中之道。

## 问题 4：能否用纯文本（Markdown）传输数据代替 JSON？需要多次循环吗？

**Skill 输出适合用 Markdown（同时附带结构化字段供 Tool 调用），Tool 之间必须保持结构化传输。不需要额外增加循环次数。**

### 分层次分析

**Skill 输出 → 用户展示：Markdown 天然适合**

问题建模 Skill 输出的 SAO 列表、因果链、矛盾描述，本身就是半结构化的自然语言内容。Skill 可以在一次输出中同时生成：
- 用户可读的 Markdown（CLI 直接渲染，无需额外渲染层）
- 隐藏的结构化字段（供下游 Tool 调用，如 `contradiction_desc`、`problem_type`）

方案生成和评估同理。

**Tool 输入输出 → 必须结构化**

`contradiction_solver` 需要精确的 `problem_type` 和 `contradiction_desc` 做参数映射，从 Markdown 中解析这些值不可靠。`convergence_control` 需要 `max_ideality` 浮点数做阈值比较，正则抠文本里的数字有风险。`cross_domain_search` 需要准确的原理 ID 列表查数据库。

**混合方案**：

```
[问题建模 Skill]
  → 输出 Markdown（用户看）+ 附带结构化字段（Tool 用）
  
[矛盾求解 Tool]
  → 输入：结构化参数（problem_type, contradiction_desc）
  → 输出：结构化对象（principles, match_conf）
  
[跨界检索 Tool]
  → 输入：结构化参数（principles, domain_hint）
  → 输出：结构化对象（cases）
  
[方案生成 Skill]
  → 输入：结构化数据 + Markdown 案例描述
  → 输出：Markdown（用户看）+ 结构化字段
  
[收敛控制 Tool]
  → 输入：结构化参数（max_ideality, unresolved_signals）
  → 输出：结构化决策
```

### 是否需要更多循环？

不需要。数据传输方式（纯文本 vs JSON）不影响 DAG 的执行步数。Skill 在一次调用中同时产出 Markdown 和结构化字段，与只产出 JSON 再额外渲染相比，调用次数完全一样。只是输出格式从单一 JSON 变成 JSON + Markdown 并行的富输出。

### 若强制全链路纯文本

意味着每个 Tool 在收到 Markdown 文本后都需要做信息提取（本质上是微型 NER 任务），这要么增加脆弱的正则逻辑，要么给每个 Tool 入口加一次 LLM 调用做文本解析，反而增加复杂度和调用次数。不推荐。

# 问题建模技能

## 目标
将用户的原始问题描述，转化为结构化的矛盾描述，为后续矛盾求解做准备。

## 输入
- `question: str` - 用户原始问题
- `resources: dict` - 可用资源（从会话上下文获取）

## 工作流程

### 步骤1：功能建模 (M1)
从用户问题中提取 S-A-O 三元组，标记功能类型，盘点资源，声明 IFR。

**调用脚本**: `scripts/m1_modeling.py --question "$question"`

**输出**:
- `sao_list`: SAO 对象列表
- `resources`: 资源字典
- `ifr`: 理想最终结果

**边界处理**:
- 若无法提取任何 SAO，返回 CLARIFY 并提示用户补充信息

### 步骤2：根因分析 (M2)
从 SAO 中的负面功能出发，构建因果链，定位根因节点，提取候选物理属性。

**门控条件**: 仅在存在 Harmful 或 Insufficient 类型 SAO 时执行

**调用脚本**: `scripts/m2_causal.py --sao-list "$sao_list"`

**输出**:
- `root_param`: 根因参数
- `key_problem`: 关键问题
- `candidate_attributes`: 候选物理属性列表

**Fallback**: 若因果链无法收敛，使用第一个 Harmful SAO 构造简化结果

### 步骤3：问题定型 (M3)
从根因中提取矛盾类型和矛盾描述。

**调用脚本**: `scripts/m3_formulation.py`

**输出**:
- `problem_type`: "tech" 或 "phys"
- `contradiction_desc`: 矛盾自然语言描述

## 🔧 编排器如何加载和调用 Skills

```python
# orchestrator.py
from pathlib import Path
from typing import Dict
import yaml

class Orchestrator:
    def __init__(self, skills_root: Path):
        self.skills_root = skills_root
        self.skills_metadata: Dict[str, dict] = {}
        self._load_skills_metadata()
    
    def _load_skills_metadata(self):
        """启动时加载所有技能的元数据（name + description）"""
        for skill_dir in self.skills_root.iterdir():
            if not skill_dir.is_dir():
                continue
            skill_md = skill_dir / "SKILL.md"
            if not skill_md.exists():
                continue
            
            # 读取 YAML frontmatter
            content = skill_md.read_text(encoding="utf-8")
            if content.startswith("---"):
                _, frontmatter, _ = content.split("---", 2)
                metadata = yaml.safe_load(frontmatter)
                self.skills_metadata[metadata["name"]] = {
                    "description": metadata["description"],
                    "path": skill_dir,
                    "full_content": content  # 不预先加载完整内容到上下文
                }
    
    def use_skill(self, skill_name: str, ctx: WorkflowContext) -> dict:
        """调用指定的 Skill，加载完整 SKILL.md 并执行"""
        skill = self.skills_metadata.get(skill_name)
        if not skill:
            raise ValueError(f"Skill '{skill_name}' not found")
        
        # 1. 注入完整 SKILL.md 到 LLM 上下文
        # 2. 让 LLM 按照 SKILL.md 中的工作流程执行
        # 3. 如果需要运行 scripts/ 中的脚本，通过 Tool 调用
        
        return self._execute_skill(skill, ctx)
```

```
tools/           → 每个 Tool 一个文件
```

这种不对称反而合理：**Tool 是确定性函数，通常只需要一个文件加数据库查询**；**Skill 有 LLM 调用、prompt 管理、输出 schema、测试等更复杂的内容**，需要更大的物理空间承载。

### 5. 未来扩展性

当 Skill 数量增长时，方案 A 的目录结构自然扩展：

```
skills/
├── problem_modeling/
├── solution_generation/
├── solution_evaluation/
├── patent_analysis/        # 未来新增：专利分析 Skill
├── knowledge_extraction/   # 未来新增：知识提取 Skill
└── registry.py
```

方案 B 在 Skill 超过 5 个后文件数爆炸，`schemas.py` 变成上千行的怪物文件。

---

## registry.py 的作用

无论哪种方案，都需要一个注册表，让 Agent 或编排器知道有哪些 Skill 可用、如何调用：

```python
# skills/registry.py
from skills.problem_modeling import ProblemModelingSkill
from skills.solution_generation import SolutionGenerationSkill
from skills.solution_evaluation import SolutionEvaluationSkill

SKILL_REGISTRY = {
    "problem_modeling": ProblemModelingSkill(),
    "solution_generation": SolutionGenerationSkill(),
    "solution_evaluation": SolutionEvaluationSkill(),
}

def get_skill(name: str):
    return SKILL_REGISTRY.get(name)
```

Agent 通过注册表发现和调用 Skill，不直接硬编码 import 路径。

---

## 总结

**推荐一个 Skill 一个文件夹**。Skill 比 Tool 复杂得多（prompt 管理、schema 定义、LLM 调用策略、独立测试），物理隔离换来的是独立迭代能力、更好的可维护性和更清晰的职责边界。这只是多了一层目录，收益远大于成本。

● 我认真读了这些材料和三个模型的回答。下面是我的独立分析：

  ---
  提问1：四个架构问题

  1. M1-M3 可以合并为一个 Skill 吗？

  不建议合并。

  DeepSeek 的核心论据是"信息折损"——M1 的推理过程 M2 看不到。但这个论断是错误的：

  当前 M1 输出被完整写入 WorkflowContext，M2 接收的是完整的 ctx.model_dump()（包含 question + sao_list + resources + ifr
   + history）。M3 同样看到 M1 和 M2 的全部输出。不存在信息折损。

  真正不能合并的原因是门控的确定性：

  - 当前 M2 是否触发由 should_trigger_m2() 工具精确控制（检查是否存在 Harmful SAO）
  - 如果合并为单个 Skill，这个判断必须交给 LLM 做
  - LLM 对"是否需要根因分析"的判断不可靠，可能该跳的时候没跳，不该跳的时候跳了

  另外，M1/M2/M3 可以分别配置不同模型（M1 用轻量模型，M2 用推理模型），合并后失去这个灵活性。

  折中方案：如果一定要减少 API 调用，只合并 M2+M3（根因分析 + 问题定型），M1 保持独立。因为 M2 的门控可以在 M1
  完成后由编排器判断，M2/M3 之间没有门控。

  ---
  2. 当前编排逻辑与 Claude Code 等主流 Agent 一样吗？

  底层机制相似，控制流完全不同。

  ┌────────────────┬─────────────────────────┬────────────────────────────┐
  │      维度      │     当前 TRIZ 系统      │        Claude Code         │
  ├────────────────┼─────────────────────────┼────────────────────────────┤
  │ 控制流决策者   │ 编排器（Python 硬编码） │ LLM 自身                   │
  ├────────────────┼─────────────────────────┼────────────────────────────┤
  │ Skill 调用方式 │ 编排器直接调用函数      │ LLM 自主决定调用哪个 Skill │
  ├────────────────┼─────────────────────────┼────────────────────────────┤
  │ 迭代逻辑       │ 编排器的 while 循环     │ LLM 的思考-行动-观察循环   │
  └────────────────┴─────────────────────────┴────────────────────────────┘

  但 DeepSeek 忽略了一个关键区别：当前系统的"Skill"不是真正的 Agent Skill，而是"LLM Prompt 模板"。

  真正的 Agent Skill 应该：
  1. 有独立的输入输出 schema
  2. 可以被动态发现和调用
  3. 内部包含完整的执行逻辑

  当前系统的 Skill 只是 .md 文件作为 system prompt，执行逻辑完全在 orchestrator 和 skill_runner 中。这更像"基于 prompt
  模板的工作流引擎"，而不是"基于 Skill 的 Agent 系统"。

  这个区别很重要，因为它决定了你的系统不能简单地"像 Claude Code 那样让 LLM 自主调度"——你的 Skill 没有自包含的执行能力。

  ---
  3. 可以改为单 Agent + Skills + Tools 吗？

  当前系统已经是"准单 Agent"，但改为 LLM 驱动的单 Agent 对 TRIZ 不合适。

  当前编排器就是 Agent 的决策核心，所有 Skill 和 Tool 都是它的扩展能力。这本身就是单 Agent 架构。

  但 DeepSeek 提议的"让 LLM 自主决定调用哪个 Skill"对 TRIZ 不适用：

  TRIZ 是严格方法论，阶段顺序是方法论的一部分：
  - 必须先功能建模（M1），再根因分析（M2），再问题定型（M3）
  - 不能跳过 M2 直接做 M3，也不能先求解再建模

  如果让 LLM 自主决策，它可能：
  - 跳过根因分析（"用户问题很简单，不需要"）
  - 先做方案生成再做问题定型（本末倒置）
  - 这违背了 TRIZ 方法论的价值

  Claude Code 适合开放式任务（编程、调试），TRIZ 适合固定流程任务。用 LLM 自主决策做
  TRIZ，就像让司机自己决定考试路线——失去了标准化评估的意义。

  值得探索的方向：保留当前硬编码编排器，但在其上增加一个轻量"元 Agent"：
  - 元 Agent 负责：输入分类、判断是否需要澄清、识别非工程问题
  - 一旦确定是工程问题，交给编排器执行固定流程
  - 这样上层有灵活性，下层有确定性

  ---
  4. 模块间数据可以用纯文本传输吗？

  内部传输必须保留 JSON。DeepSeek 的"混合方案"在实践中不可行。

  DeepSeek 提议 Skill 同时输出 Markdown + 结构化字段。但 LLM 很难在一次调用中同时输出两种格式而不混淆：

  LLM 输出：
  ## 功能建模
  - 主体：刀片
  - 动作：切割

  {"sao_list": [...]}  ← 突然插入 JSON，格式极易出错

  Function Calling 强制要求 JSON：当前 M4 调用 map_to_parameters、query_matrix 等 Tool，参数必须是 JSON
  Schema。如果内部传输改为纯文本，每个 Tool 调用前都需要额外做一次"文本解析"（要么正则，要么再调一次
  LLM），反而增加复杂度和失败率。

  当前分层架构是正确的：
  - 数据层：JSON（Skill 输出、Tool 参数、WorkflowContext）
  - 展示层：Markdown（CLI 渲染给用户看）

  DeepSeek 说"主流 Agent 内部也是结构化对象"，这个说法是对的。不要为了解决"渲染层"的问题而破坏"数据层"的稳定性。

  ---
  提问2：SKILL.md 是一个 skill 一个文件夹吗？

  当前阶段不需要。

  DeepSeek 推荐文件夹结构的理由是"物理隔离 = 独立迭代"。但这个论据在当前系统不成立：

  当前每个 Skill 只有一个 .md 文件（system prompt），没有：
  - 独立的 Python 脚本（代码在 skill_runner.py）
  - 独立的 schema（在 context.py）
  - 独立的测试（在 tests/）
  - 外部资源

  为一个 .md 文件创建文件夹是过度设计。

  什么时候需要文件夹？
  当某个 Skill 变得复杂，需要：
  - 多个 prompt 变体（如中英文版本）
  - 独立的 Python handler
  - 外部数据文件（如评估标准、示例库）

  那时可以为该 Skill 单独创建文件夹。但现在没有必要为所有 Skill 平铺文件夹。

  文件名可以保留当前格式（m1_modeling.md），因为：
  - 文件名有语义（m1 = 模块编号，modeling = 功能）
  - 与 orchestrator 调用 ("m1_modeling", "Skill") 一致

  如果未来要对接外部 Agent 框架（如 Anthropic 的 Skills 规范），再考虑统一重命名为 SKILL.md 并放入文件夹。
