## 三、发现的问题

### 🔴 严重：M6 评估无法识别非工程问题

**现象**：对抗性输入（"今天天气怎么样"、"如何成为亿万富翁"等）系统仍然生成了完整方案，且 M6 的 `problem_relevance_score` 给到了 5/5 的高分。

**根因分析**：
- M1 建模阶段将这些输入解析出了 SAO 三元组（如"天气"→"影响"→"出行"）
- M3 矛盾定型提取出了"矛盾描述"
- M4 根据矛盾描述匹配到了发明原理
- M5 生成了方案草稿
- **M6 评估时没有正确识别方案与原始问题的相关性**，`problem_relevance_score` 虚高

**影响**：用户可能收到与问题无关的"幻觉"方案，降低系统可信度。

**建议**：
1. 在 M1 建模后增加一道"问题相关性检查"：判断用户输入是否是可工程化的问题
2. 或者在 M6 的 problem_relevance_score 评估中，明确要求模型将方案与用户原始问题直接对比，而不是与中间提取的矛盾对比
3. 在 M6 Skill 的 system prompt 中增加更多对抗性示例，训练模型识别不相关方案

### 🟡 中等：无效输入处理耗时过长

**现象**：古诗、纯数字、emoji 等明显无效输入仍走完了完整 workflow（100-400 秒）。

**建议**：在 M1 建模后增加快速退出路径：如果 SAO 列表质量极低（如所有 SAO 的 action 都是无意义动词），直接触发 clarify，不进入 M2-M7。

### 🟢 轻微：个别问题耗时偏长

**现象**：建筑物抗震问题平均 232.6 秒，比其他问题慢约 30%。

**建议**：监控各节点耗时，定位瓶颈（可能是 M5/M6 的模型响应慢）。

### 🟢 已修复：CLI skipped 标记未传递

**现象**：M2 gate skip 时，CLI 未正确显示"跳过"状态。

**修复**：`triz/cli.py` `_on_event` 方法中，`step_complete` 时将 `result["skipped"]` 复制到 `step["skipped"]`。

## 四、优化建议

### 短期（1-2 天）
1. **修复 M6 relevance_score 评估**：在 M6 Skill 的 system prompt 中增加"问题-方案匹配验证"的强调，要求模型直接对比用户原始问题与方案的相关性
2. **M1 快速退出**：增加对 SAO 质量的启发式检查，过滤明显无效输入

### 中期（1-2 周）
3. **API 超时重试**：当前 API client 无重试逻辑，网络波动会导致 workflow 中断
4. **耗时监控**：为每个节点增加耗时记录，便于后续优化

### 长期
5. **对抗性输入数据集**：建立一组已知会误导系统的对抗性输入，作为回归测试用例
6. **M6 评估质量验证**：考虑引入第三方模型（不同于 M6 的 Kimi-K2.5）做方案质量交叉验证
