# TRIZ 方法流程

---

## 1. 为什么需要这个系统

### 1.1 通用大模型做 TRIZ 的两个致命问题

**问题一：幻觉严重，参数选不准**

通用大模型直接问 TRIZ 问题时，会乱选 39 个工程参数：
- "速度"可能是 #9 Speed，也可能是 #19 运动物体的能量消耗
- "温度"可能是 #17 Temperature，也可能是 #22 能量损失

**问题二：只会给原理定义，不会给可落地方案**

通用大模型输出："请使用原理 19：周期性动作"
用户问："然后呢？怎么做？"
模型无法提供具体的、可执行的跨行业案例。

## 2. 整体架构概览

## 3. Phase 1：功能建模

### 3.1 目标

不仅仅"听懂"问题，而是建立系统的"电路图"，为后续"矛盾识别"打地基。

### 3.2 Step 1.1：S-A-O 提取

**什么是 S-A-O？**

S-A-O = Subject（主体）- Action（动作）- Object（客体）

这是 TRIZ 功能分析的基础模型，描述系统中组件之间的相互作用。

**示例拆解**：

| 用户描述 | Subject | Action | Object |
|---------|---------|--------|--------|
| "刀片切纸" | 刀片 | 切割 | 纸张 |
| "刀片磨损" | 纸张 | 磨损 | 刀片 |
| "摩擦生热" | 刀片 | 摩擦 | 纸张 |

### 3.3 功能属性标记详解

**三种功能类型**：

| 类型 | 定义 | 标记 | 示例 |
|------|------|------|------|
| **有用功能** | 达成系统目标所需的功能 | `Useful` | 刀片切割纸张 |
| **不足功能** | 有用但性能不够的功能 | `Insufficient` | 刀片切割太慢 |
| **有害功能** | 对系统产生负面效果的功能 | `Harmful` | 刀片磨损、摩擦生热 |

**功能属性标记有什么用？**

1. **指导 Phase 3 的求解路径选择**
   - 有害功能 → 优先尝试 Path B（物理矛盾转化）
   - 不足功能 → 标准 Path A（技术矛盾）

2. **Phase 5 方案评估的权重调整**
   - 有害功能多的问题，理想度公式中"有害副作用"权重增加

3. **资源盘点的优先级排序**
   - 有害功能相关的组件，优先作为"可裁剪"候选

### 3.4 Step 1.2：资源盘点

**TRIZ 中的六大资源类型**：

| 资源类型 | 子类别 | 示例 |
|---------|--------|------|
| **物质资源** | 系统内物质 | 刀片、纸张、废料 |
| | 超系统物质 | 空气、水、土壤 |
| **场资源** | 物理场 | 重力场、磁场、电场 |
| | 能量场 | 热场、光场、声场 |
| **空间资源** | 空腔/空隙 | 间隙、孔洞、夹层 |
| | 表面/界面 | 接触面、边界层 |
| **时间资源** | 动作前/后 | 准备时间、冷却时间 |
| | 动作间隙 | 停顿、周期 |
| **信息资源** | 测量信息 | 温度、压力、速度数据 |
| | 知识信息 | 专利、论文、案例 |
| **功能资源** | 已有功能 | 现有组件的附加功能 |

**资源注入 Phase 5 的具体作用**：

**作为"免费资源"降低方案成本**

```python
synthesis_prompt = f"""
生成创新方案时，必须优先考虑使用以下已有资源：
{resources}

原则：
- 优先使用系统内资源，避免引入新组件
- 利用超系统资源（环境）降低成本
- 利用场资源实现功能替代

示例：
如果用户有"热场"资源，优先考虑热切割而非机械切割。
"""
```

## 4. Phase 2：问题建模

### 4.1 目标

彻底解决 LLM 无法准确对应"39个工程参数"的幻觉问题。

### 4.2 RCA+ 因果链分析

**什么是 RCA+？**

RCA = Root Cause Analysis（根本原因分析）  
RCA+ = 增强版根本原因分析，结合因果链（Cause-Effect Chain）方法

**核心目标**：不只是找到表面原因，而是追溯到问题的**根因节点（Root Cause Node）**

**为什么需要 RCA+？**

| 问题 | 示例 |
|------|------|
| **表面问题** | "刀片磨损太快" |
| **直接原因** | "摩擦太大" |
| **根因节点** | "接触面积过大导致热积累" |

**RCA+ 的作用**：
1. **根因才是矛盾的真正所在**（接触面积既要大又要小）
2. **根因决定了物理属性的提取方向**（热、面积、压力）
3. **根因指导 Phase 3 的物理矛盾转化**

**RCA+ 具体示例**：

**用户问题**："手术刀片切割组织时磨损太快，需要频繁更换"

**因果链分析**：

```
┌─────────────────────────────────────────────────────────────┐
│                      因果链分析                               │
├─────────────────────────────────────────────────────────────┤
│  Level 0 (表面问题)                                         │
│  └── 刀片磨损太快 ← 用户直接描述                              │
│                                                             │
│  Level 1 (直接原因)                                          │
│  └── 为什么磨损快？→ 摩擦产生的热量过高                      │
│      └── 为什么热量高？→ 接触面积大 + 切割时间长              │
│                                                             │
│  Level 2 (深层原因)                                          │
│  └── 为什么接触面积大？→ 刀片设计需要足够宽度保证强度         │
│      └── 为什么切割时间长？→ 组织阻力大，需要持续施力         │
│                                                             │
│  Level 3 (根因节点) ⭐                                        │
│  └── 【根因 1】接触面积必须大（为了强度/切断）               │
│  └── 【根因 2】接触面积必须小（为了减少摩擦热/磨损）         │
│                                                             │
│  → 识别出物理矛盾：接触面积【既要大又要小】                   │
└─────────────────────────────────────────────────────────────┘
```

```

**RCA+ 如何影响后续步骤？**

1. **指导物理属性提取（Step 1）**

```python
# 没有 RCA+ 时，LLM 可能提取的表面属性：
attributes = ["速度", "硬度"]  # 太泛，不准确

# 有 RCA+ 后，精准提取根因相关属性：
attributes = ["接触面积", "热积累", "摩擦系数", "切割强度"]
```

## 5. Phase 3：神经符号求解

### 5.1 目标

既要标准解（技术矛盾），更要突破解（物理矛盾）。

### 5.2 阿奇舒勒矛盾矩阵是什么？

**定义**：由 TRIZ 创始人阿奇舒勒（Altshuller）创建的 39×39 查询表，用于解决**技术矛盾**。

**技术矛盾定义**：改善某个参数时，导致另一个参数恶化。

**矩阵结构**：
- 行：要改善的参数（1-39）
- 列：恶化的参数（1-39）
- 单元格：推荐的发明原理编号列表（1-40）

**示例**：
```
改善 #9 Speed（速度） → 恶化 #12 Shape（形状稳定性）
查矩阵 Matrix[9][12] = [15, 28, 35]
推荐原理：#15 动态化、#28 机械系统替代、#35 参数变化
```


### 5.3 Path B：物理矛盾

**适用场景**：同一个参数既要大又要小（既要强度又要轻量）

**实现方式**：

1. **转化 Agent**：基于 RCA+ 因果链分析结果，识别"既要...又要..."节点

2. **判定分离类型**

3. **查分离规则库**

## 6. Phase 4：FOS跨界检索

### 6.1 目标

提供"他山之石"，而非简单的定义解释。

### 6.2 核心逻辑：FOS（Function-Oriented Search）

**传统 RAG 的问题**：
- 用户问："怎么修眼镜？"
- 搜出来的是眼镜维修教程，没有创新

**FOS 的做法**：
- 构造查询：`Principle_ID=14` AND `Function="增加刚性"` AND `Context="轻量化结构"`
- 搜出来的是：F1 赛车单体壳结构、航天器蜂窝结构、医疗器械增强方案

### 6.3 Query 构造对比

| 普通搜索 | FOS 搜索 |
|---------|---------|
| "怎么修眼镜" | `Principle_ID=14` AND `Function="增加刚性"` AND `Context="轻量化结构"` |
| "刀片磨损怎么办" | `Principle_ID=28` AND `Function="减少摩擦"` AND `Resource="热场"` |

### 6.5 检索源

预处理过的专利/案例向量库（包含 Tag：医疗、航天、半导体）


### 6.6 预期效果示例

> "参考 F1 赛车的单体壳结构（原理14+17），解决眼镜腿易断问题。"

## 7. Phase 5：具象化与评测

### 7.1 目标

生成高价值方案，并量化其质量。

### 7.2 Step 5.1：方案合成（Synthesis Agent）

**输入**：
- 原理 ID（来自 Phase 3）
- 跨界案例（来自 Phase 4）
- 可用资源（来自 Phase 1）

**约束**：必须使用"类比法"将案例迁移到用户场景

### 7.3 Step 5.2：理想度打分（Critic Agent）

**公式**：

```
Ideality = (0.5 × 功能突破 + 0.3 × 专利潜力) / (0.15 × 成本 + 0.05 × 复杂度)
```

**权重策略**：侧重 **功能突破 (50%)** 和 **专利潜力 (30%)**
