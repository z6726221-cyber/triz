# M5 方案生成

## 任务
将抽象的发明原理和跨界案例迁移到用户的具体场景，生成具体可执行的方案草稿。

## 输入
你会收到用户问题和之前的分析结果（包括矛盾描述、发明原理、IFR、资源等）。基于这些信息生成方案。

## 四步流程

### 第一步：生成搜索词
基于问题、矛盾和原理，生成 3 个不同角度的英文搜索词：
1. **功能角度**：聚焦核心功能的专利
2. **原理角度**：聚焦发明原理的跨领域应用
3. **问题角度**：聚焦用户的具体问题

每个搜索词 3-8 个英文关键词，用空格分隔。不要用中文（Google Patents 不支持中文搜索）。

### 第二步：过滤搜索结果
如果系统返回了搜索结果，评估每个结果与用户问题的相关性（1-5分），只保留相关性 >= 3 的结果。

### 第三步：提取关键模式
从过滤后的案例中，提取 2-3 个可迁移的工程模式。

### 第四步：生成方案
基于发明原理、过滤后的案例和关键模式，生成具体方案。

详细的流程说明、示例和约束，请参考 `references/generation_guide.md`。

## 输出格式

请输出以下 Markdown 内容：

### 搜索词

1. （功能角度）
2. （原理角度）
3. （问题角度）

### 关键模式

- 模式1
- 模式2

### 方案

#### 方案 1：（标题）

**应用原理**：原理 X（名称）

**方案描述**：（至少100字的具体技术方案描述）

**资源映射**：（使用了哪些现有资源）

#### 方案 2：（标题）

...

## 约束

1. 每个方案必须明确引用一个或多个发明原理编号
2. 方案必须具体、可执行，避免泛泛而谈（至少100字描述）
3. 优先使用用户已有的资源，避免引入全新组件
4. 如果无法生成方案，明确说明原因

## Gotchas（常见陷阱）

1. **搜索词用中文**：Google Patents 不支持中文搜索 → 搜索词必须是英文关键词
2. **方案未引用原理**：LLM 生成的方案容易泛泛而谈 → 每个方案必须明确引用发明原理编号
3. **方案过于简短**：LLM 倾向于一句话方案 → 描述至少 100 字，包含具体的技术实现细节
4. **忽略已有资源**：方案应优先使用用户已有的资源（来自 M1 的资源盘点），避免引入全新组件
5. **重复已知原理**：方案只是复述原理本身而非具体应用 → 应结合用户场景做迁移
